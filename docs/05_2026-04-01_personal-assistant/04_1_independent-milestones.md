# 个人助理独立 Milestone 体系设计

> 基于 2026-04-02 当前代码与文档状态，对 `docs/05_2026-04-01_personal-assistant/` 现有方案补充一个**独立于主线 M0~M12 的产品线 milestone 体系**。
>
> 结论先行：
> - `M0 ~ M12` 继续作为 **NeuroCore 引擎 / 平台能力里程碑**
> - `PA-M1 ~ PA-M6` 作为 **Personal Assistant 产品线里程碑**
> - `03_implementation-plan.md` 中的 `M-PA-1 ~ M-PA-11` 调整为 **工程工作包 / 实施任务**，用于支撑产品 milestone，而不是替代产品 milestone 本身

---

## 1. 为什么要独立于主 milestone

当前主 milestone 的语义是“认知引擎能力闭环”：

- `M8` 聚焦世界模型与设备接入
- `M9` 聚焦多 Agent 调度
- `M10` 聚焦技能强化学习
- `M11` 聚焦 Console
- `M12` 聚焦通用自主体

但个人助理是一个**垂直产品线**，它的交付边界不是某一个底层模块，而是一个面向真实用户的完整体验闭环。直接把个人助理排进主 milestone，会出现三个问题：

1. **排期语义混淆**
   主 milestone 解决的是底座能力是否存在；个人助理要解决的是用户今天能不能真的用起来。

2. **依赖关系失真**
   个人助理会消费 `memory / policy / multi-agent / world-model` 等多条能力线，但并不要求 `M10 / M11 / M12` 全部完成后才能交付产品增量。

3. **验收标准不对齐**
   主 milestone 的验收更偏“系统能力是否闭环”；个人助理的验收更偏“一个用户场景是否能稳定跑通并被长期使用”。

因此，个人助理需要一套**以用户价值、真实交互路径和可运营闭环为中心**的独立 milestone 体系。

---

## 2. 设计原则

### 2.1 主线与产品线分层

- 主 milestone 负责交付可复用的引擎能力
- 个人助理 milestone 负责把这些能力切成可发布的产品能力
- 两者是“底座 / 产品”关系，不是“父子里程碑”关系

### 2.2 以垂直切片而不是技术模块分期

每个 `PA-M*` 都必须形成一条可演示、可测试、可追踪的用户闭环，而不是只完成一个技术组件。

### 2.3 只把通用能力下沉到底座

如果某能力只对个人助理有价值，就留在 `im-gateway / proactive-engine / service-connectors` 这一层；只有当它被证明是跨产品通用能力时，才考虑下沉到主线包。

### 2.4 默认复用现有引擎，不重新发明核心运行时

个人助理层默认复用：

- `runtime-core` 的 session / cycle / checkpoint / approval / trace
- `memory-core` 的 working / episodic / semantic / procedural memory
- `multi-agent` 的 delegation / registry / bus
- `world-model` / `device-core` 的世界状态与设备 SPI
- `sdk-core` / `sdk-node` 的 agent 组装能力

### 2.5 每个 milestone 都要可运营

每个 `PA-M*` 收口时都至少具备：

- 一个真实用户场景
- 一条可回放 trace
- 一组最小 E2E 测试
- 明确的审批、失败恢复和观测路径

## 3. 与主 milestone 的关系

### 3.1 主 milestone 的角色：能力底座，不是排期容器

| 主 milestone | 对个人助理的意义 | 是否为当前阻塞 |
|---|---|---|
| M1 ~ M7 | 运行时、记忆、工具、评估、托管基础 | 否，已具备 |
| M8 | 设备 / 世界模型底座 | 非 Phase A 阻塞，后续能力可消费 |
| M9 | 动态委派与子 Agent 底座 | 非基础对话阻塞，是复杂任务的关键底座 |
| M10 | 技能强化学习优化层 | 非阻塞，可后置 |
| M11 | Console 运维界面 | 非阻塞，只影响观测体验 |
| M12 | 通用自主体 | 非阻塞，未来可局部吸收其规划能力 |

### 3.2 个人助理自有新增层

个人助理产品线明确拥有以下新增组件：

- `im-gateway`
- `proactive-engine`
- `service-connectors`
- `assistant composition / worker catalog / notification policy`
- `cross-channel identity & conversation routing`

这些组件不属于 M8~M12 原始定义，应该由个人助理产品线自己负责排期和收口。

### 3.3 对 `03_implementation-plan.md` 的重新解释

`03_implementation-plan.md` 中的 `M-PA-1 ~ M-PA-11` 保留，但语义调整为：

- **工作包（work packages）**
- 用于实现某个 `PA-M*` 所需的工程任务
- 允许一个 `PA-M*` 由多个 `M-PA-*` 共同组成
- 允许一个 `M-PA-*` 同时服务多个 `PA-M*`

换句话说：

- `PA-M*` 回答“产品本轮要交付什么”
- `M-PA-*` 回答“工程上这轮具体做哪些包和测试”

## 4. Personal Assistant 独立里程碑总览

### 4.1 里程碑图

```text
PA-M1 对话入口 MVP
   ↓
PA-M2 可执行事务助手
   ↓
PA-M3 主动运营闭环
   ↓
   ├── PA-M4 个人知识与记忆
   └── PA-M5 复杂任务委派
            ↓
         PA-M6 环境化多设备助手
```

说明：

- `PA-M1 → PA-M3` 是当前最短商业闭环
- `PA-M4` 与 `PA-M5` 可在 `PA-M3` 后并行推进，但默认建议先做 `PA-M4`
- `PA-M6` 是环境化扩展层，不进入当前最短路径

### 4.2 里程碑表

| 里程碑 | 定位 | 目标结果 |
|---|---|---|
| `PA-M1` | 对话入口 MVP | 用户可以通过飞书 / Web Chat 稳定和助理对话，并获得实时搜索能力 |
| `PA-M2` | 可执行事务助手 | 助理可以安全读取并操作邮件 / 日历等个人事务，并带审批闭环 |
| `PA-M3` | 主动运营闭环 | 助理可以周期性检查、触发主动提醒、执行定时摘要 |
| `PA-M4` | 个人知识与记忆 | 助理具备用户偏好、文档知识库和可治理的长期记忆能力 |
| `PA-M5` | 复杂任务委派 | 助理能把复杂任务拆成可追踪的子任务并动态委派执行 |
| `PA-M6` | 环境化多设备助手 | 助理接入设备与环境事件，形成更持续的 ambient assistant |

## 5. 各 milestone 详细设计

### PA-M1：对话入口 MVP

> 2026-04-25 校准：PA-M1 本地 Web Chat 连续性完成度提升到约 92%。当前已补齐终态 runtime session 重开时的 `conversation_handoff`，同一 chat 的最近用户/助手消息会进入新 session 初始输入 metadata。

**产品目标**：
建立一个真正可用的个人助理入口，而不是“能跑 demo 的 runtime example”。

**用户可感知能力**：

- 在飞书向助理发消息并收到回复
- 在 Web Chat 本地调试同一能力
- 助理可以调用搜索 / 浏览器工具回答实时问题
- 基本 chat thread 连续性成立，runtime session 完成后重开也能携带最近对话上下文，支持 `/new`、`/reset`

**核心范围**：

- `IM Gateway`
- 飞书 Adapter
- Web Chat Adapter
- `ConversationRouter`
- Assistant 组装
- `web_search` / `web_browser`
- 基础 trace / approval / command 接线

**明确不做**：

- 邮件 / 日历写操作
- 主动提醒
- 真正的多 Agent 复杂委派
- 知识库上传
- 多设备联动

**退出标准**：

- 飞书与 Web Chat 两个入口都可稳定收发消息
- 至少 1 条实时搜索问答链路可端到端跑通
- 新建 / 重置 / 超时重开 session 行为明确，终态 session 重开不丢同一 chat 的短指代上下文
- 失败时能在 trace 中定位到入口、工具和响应链路

**建议对应工作包**：

- `M-PA-1`
- `M-PA-2`
- `M-PA-3` 的 Phase A 子集

---

### PA-M2：可执行事务助手

**产品目标**：
把助理从“会聊、会搜”升级成“能安全做事”的事务代理。

**用户可感知能力**：

- 查询日程、收件箱、最近邮件
- 起草并发送邮件，但发送前必须审批
- 创建 / 修改日程，并按风险等级门控
- 查询当前执行状态、历史 session、当前目标摘要

**核心范围**：

- `email_read` / `email_send`
- `calendar_read` / `calendar_write`
- 审批卡片 / 审批恢复
- 命令体系（`/status`、`/history`、`/skills`）
- 跨 session 的事务状态恢复

**关键设计点**：

- 这一 milestone 的本质不是“接更多 connector”，而是形成**安全执行闭环**
- 高副作用动作必须默认接入 Amygdala 审批链，而不是在产品层单独写旁路逻辑
- 任何事务执行都要保留审计和中断能力

**退出标准**：

- 邮件读取与日历读取稳定可用
- 邮件发送必须经过审批且审批后可恢复执行
- 日历写入具备风险分级与 trace 记录
- 用户可以查看当前 session / goal / 历史摘要

**建议对应工作包**：

- `M-PA-5`
- `M-PA-6`
- `M-PA-7` 的事务场景子集

---

### PA-M3：主动运营闭环

**产品目标**：
让助理从“只响应消息”变成“能主动发现值得通知的事情”。

**用户可感知能力**：

- 每日摘要
- 邮件 / 日历变化提醒
- 定时提醒事项
- 不同优先级的通知推送

**核心范围**：

- `ProactiveEngine`
- `HeartbeatScheduler`
- `CronScheduler`
- `EventSource` SPI
- Notification priority / quiet hours / basic dedup

**关键设计点**：

- 主动行为必须通过 session 跑 NeuroCore，而不是在调度器里直接拼通知文本
- 主动通知是“候选行为”，仍然要经过记忆、策略、风险和上下文整合
- 要防止心跳把系统变成“定时刷屏器”，因此必须有频率控制和去重

**退出标准**：

- 至少 2 类心跳检查可触发主动通知
- 至少 1 个 cron 定时任务可稳定运行
- 所有主动通知都能追溯到触发源、会话、判定理由和发送记录
- 基本去重与静默策略可用

**建议对应工作包**：

- `M-PA-4`
- `M-PA-5` 的读取能力部分
- `M-PA-7` 的主动场景子集

---

### PA-M4：个人知识与记忆

> 2026-04-25 增量：用户偏好记忆的最小治理闭环已落地。`examples/personal-assistant/src/memory/` 新增 SQLite-backed `PersonalMemoryStore`，IM 命令支持 `/remember`、`/forget`、`/correct`、`/memories`，普通消息会把 active memories 注入 runtime input metadata。PA-M4 仍未完成文档知识库、引用式知识问答、衰减和隐私标签。

**产品目标**：
让助理真正变成“懂这个用户”的系统，而不是每轮都从零开始的聊天机器人。

**用户可感知能力**：

- 记住长期偏好和稳定习惯
- 上传文档并进行问答
- 基于历史对话与文档回答个性化问题
- 支持删除、失效和敏感信息治理

**核心范围**：

- 显式个人偏好记忆命令与 active/tombstoned 状态
- 知识库 ingest
- `knowledge_search`
- 个人偏好 profile
- 个人知识 / 对话 / 文档的统一检索层
- 基础隐私标签、删除与过期策略

**关键设计点**：

- `PA-M4` 关注的是**产品可用的个人知识面**
- 它可以先建立在现有四层记忆上，不等待五层记忆系统演进完成
- 未来 `docs/05_2026-04-01_memory-evolution/` 收口后，可把底层存储 / 检索替换掉，但不改变产品 milestone 定义

**退出标准**：

- 至少支持 PDF / Markdown / TXT 摄入
- 用户偏好可以跨 session 生效
- 知识问答能够给出引用片段或来源指示
- 提供最小删除 / 降权 / 敏感信息控制能力

**建议对应工作包**：

- `M-PA-8`
- `M-PA-11` 的知识库验证部分

---

### PA-M5：复杂任务委派

**产品目标**：
把助理升级为能处理“调研、汇总、审阅、投递”这类长链路复杂任务的执行系统。

**用户可感知能力**：

- 复杂任务自动拆解
- 进度持续推送
- 子任务并行处理
- 结果审阅、失败重试和局部重跑

**核心范围**：

- `worker catalog`
- delegation policy
- progress notification
- session chain / long-running task continuation
- verifier / reviewer / formatter 等运行时角色模板

**关键设计点**：

- `PA-M5` 复用 `M9` 已有的 delegation primitives，但产品层要补齐“怎么选 worker、怎么注入权限、怎么汇报进度”
- 子 Agent 是运行时角色，不是硬编码工作流节点
- 任何委派都要带 `goal + context slice + tool scope + budget + timeout`

**退出标准**：

- 至少 1 个复杂任务场景能稳定完成，例如“调研并发报告到邮箱”
- 子任务失败不会导致整条链路不可恢复
- 用户能看到任务阶段、当前 bottleneck 和最终汇总结果
- 委派路径有 trace，可区分主会话和 delegated session

**建议对应工作包**：

- `M-PA-3` 的委派预留部分
- `M-PA-7` 的复杂任务场景扩展
- `M-PA-9` 的技能包可作为加速层，但不是前置阻塞

---

### PA-M6：环境化多设备助手

**产品目标**：
让个人助理从“聊天入口”扩展为“常驻环境中的个人操作层”。

**用户可感知能力**：

- 文件变化、设备通知、剪贴板等环境事件进入助理
- 助理可以把结果同步到不同设备
- 同一用户在多个入口与设备之间保持连续性

**核心范围**：

- 基于 `device-core` 的 sensors / actuators
- file watcher / notification relay / clipboard sync
- 统一用户身份与设备映射
- event-driven automation

**明确不做**：

- 广义 IoT 平台化
- 复杂分布式设备协同
- 抽象成通用设备管理产品

**退出标准**：

- 至少 2 类设备或环境事件能够接入
- 至少 1 条跨设备动作链路可稳定工作
- 设备事件不会绕过审批 / trace / policy 体系

**建议对应工作包**：

- `M-PA-10`
- `M-PA-11` 的多设备验证部分

- `M-PA-*` 只作为研发任务管理粒度

## 7. 当前推荐执行顺序（2026-04-02）

### 7.1 推荐顺序

1. `PA-M1` 对话入口 MVP
2. `PA-M2` 可执行事务助手
3. `PA-M3` 主动运营闭环
4. `PA-M4` 个人知识与记忆
5. `PA-M5` 复杂任务委派
6. `PA-M6` 环境化多设备助手

### 7.2 为什么不是先做 `PA-M5`

虽然 `M9` 底座已经可用，但当前产品最短价值路径不是“先证明多 Agent 很强”，而是先证明：

- 用户愿意把助理当成稳定入口
- 助理能安全读取和执行个人事务
- 助理会主动提醒有价值的事情

在这三点没有成立之前，复杂委派的体验价值不够稳定，反而会把产品带向“炫技 demo”而不是“真实个人助理”。

### 7.3 当前阶段的最短闭环

当前最短闭环定义为：

`PA-M1 + PA-M2 + PA-M3`

只要这三个 milestone 收口，个人助理就已经具备：

- 固定入口
- 安全执行
- 主动提醒

这已经是一个可以真实持续使用的 v1 产品。

## 8. 里程碑验收模板

后续每个 `PA-M*` 收口时，统一用下面四类问题判断是否真的完成：

1. **用户价值是否闭环**
   是否有一个真实用户愿意重复使用该能力，而不是只能演示一次。

2. **系统链路是否闭环**
   是否从入口、session、工具、审批、trace、恢复一路连通。

3. **失败路径是否闭环**
   是否存在超时、失败、审批拒绝、重试和人工中断路径。

4. **观测与治理是否闭环**
   是否能看到是谁触发、为什么触发、调用了什么、为什么被放行或拦截。

如果四项里有任意一项不成立，就不应把该 milestone 记为完成。

## 9. 最终口径

个人助理不应继续附着在 `M8 ~ M12` 的主 milestone 语义之下。

更准确的表达应是：

- `M0 ~ M12` 定义 NeuroCore 底座能力的演进
- `PA-M1 ~ PA-M6` 定义 Personal Assistant 产品线的演进
- `M-PA-1 ~ M-PA-11` 只是落实这些产品 milestone 的工程工作包

这样处理后，个人助理产品线可以：

- 复用主线能力，但不被主线排期绑死
- 用用户价值驱动自己的阶段收口
- 在未来继续吸收 `memory-evolution / M10 / M12` 的成果，而不需要重写自己的 milestone 定义
