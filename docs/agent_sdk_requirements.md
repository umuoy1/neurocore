# NeuroCore Agent SDK 需求方案

## 1. 文档目的

本文档用于把 `docs/NeuroCore_Agent_Architecture.md` 中提出的 Agent 新范式，收敛为一个可对外提供的基础 SDK 与运行时产品需求方案。目标不是继续讨论“Agent 应该像大脑”，而是明确：

- 这个系统到底解决什么问题
- 对谁提供价值
- 第一版需要交付哪些能力
- 哪些能力必须进入内核，哪些可以延后
- 如何判断该方案是否成功

本文档面向产品、架构、平台、SDK、模型工程与业务落地方共同评审。

## 2. 对现有 docs 中新范式的分析

现有文档提出的核心判断是正确的：ReAct 解决了“LLM 能调用工具”的问题，但没有解决“Agent 如何形成稳定、可扩展、可积累的认知系统”。

从工程视角看，原文可以被提炼为七个问题定义：

1. ReAct 是串行链式调用，缺少并行认知与统一调度。
2. ReAct 没有世界模型，行动前无法做前向预测和风险评估。
3. ReAct 没有结构化长期记忆，无法沉淀经验、规则和技能。
4. ReAct 没有元认知层，无法做置信度监控、冲突检测和资源分配。
5. ReAct 没有动机与优先级系统，无法在复杂任务里做取舍。
6. ReAct 容错能力弱，一次错误容易沿推理链级联放大。
7. ReAct 没有层次化目标管理，难以支撑长任务、多目标和任务恢复。

现有文档的关键价值，不在于“神经科学映射”本身，而在于它给出了一个更适合工程实现的设计方向：

- Agent 不应等同于一次 LLM 调用，而应被定义为一个认知运行时。
- LLM 只负责高阶推理，不应承担记忆、调度、世界状态维护、执行控制的全部职责。
- 系统应围绕“认知周期”运行，而不是围绕“单轮 prompt”运行。
- Agent 应积累三类资产：记忆、技能、状态模型。
- Agent 的核心竞争力不只是回答质量，而是稳定执行、恢复能力、成本控制和可解释性。

这意味着 NeuroCore 的真正产品化方向应当是：

- 不是做一个“更复杂的 prompt 模板”
- 而是做一个“认知内核 + SDK + 运行时”的基础设施

## 3. 产品定位

### 3.1 产品定义

NeuroCore Agent SDK 是一个面向外部开发者、平台团队和企业客户的 Agent 基础设施产品，提供以下能力：

- 结构化 Agent 定义能力
- 认知周期编排能力
- 目标管理与状态管理能力
- 多层记忆能力
- 世界模型与预测接口
- 技能沉淀与复用能力
- 安全治理、观测、评估与回放能力

它既可以以 SDK 方式嵌入业务系统，也可以以 Runtime 方式独立部署，作为企业级 Agent 平台的基础内核。

### 3.2 产品边界

NeuroCore 不直接做以下内容：

- 不做某个垂直业务的完整应用产品
- 不绑定单一模型厂商
- 不把“提示词编排器”作为主要形态
- 不替代业务系统中的 ERP、CRM、工单、代码仓等外部系统
- 不在第一版追求完整 AGI 式通用自主体

NeuroCore 要做的是：为上层业务构建“可运行、可治理、可积累、可扩展”的 Agent 底座。

### 3.3 产品形态

产品建议采用“双形态”：

1. 嵌入式 SDK
   适合中小规模应用、单进程执行、开发者本地集成。

2. 托管式 Runtime
   适合企业级场景、异步任务、复杂编排、多租户、审计和观测要求高的环境。

## 4. 目标用户与使用角色

### 4.1 外部目标用户

1. AI 应用开发团队
   需要快速搭建客服 Agent、研究 Agent、编码 Agent、自动化 Agent。

2. 企业平台团队
   需要统一接入模型、工具、知识库、审计、权限和 Agent 运行时。

3. 垂直 SaaS 厂商
   希望把 Agent 作为其产品能力的一部分对外输出。

4. 高复杂任务团队
   例如数据分析、代码生成、运维处置、调度决策、流程自动化等。

### 4.2 内部使用角色

1. Agent Developer
   定义 Agent、工具、技能、记忆策略和执行策略。

2. Platform Operator
   负责模型接入、资源调度、审计、监控、租户治理。

3. Domain Owner
   提供业务规则、工具权限、评估指标和人类介入策略。

4. Evaluator
   负责回放、评估、对比测试与版本验收。

## 5. 核心价值主张

相较于传统 ReAct Agent 框架，NeuroCore 需要明确对外提供以下价值：

1. 更稳定
   通过目标栈、元认知、策略切换、检查点和冲突检测，降低长链任务失稳概率。

2. 更高效
   通过并行模块、技能复用、上下文压缩和按需调用，减少无效模型调用与 Token 消耗。

3. 更可积累
   通过情景记忆、语义规则、程序技能与经验回放，让系统可以随着运行持续变强。

4. 更可治理
   提供可观测、可审计、可回放、可限权、可降级的工程能力。

5. 更可扩展
   支持接入不同 LLM、向量库、图存储、工具体系、策略模块和安全模块。

## 6. 典型应用场景

### 6.1 通用复杂任务 Agent

用户给出一个结果导向的复杂目标，Agent 需要分解目标、补齐信息、制定策略、调用多个工具、跟踪状态并在失败后恢复。

典型例子：

- 代码仓库问题定位与修复
- 营销活动方案生成与执行编排
- 多数据源报表分析与归因
- 采购比价与决策建议

### 6.2 企业流程自动化 Agent

Agent 需要跨多个系统执行操作，并保证权限、审计、回滚、人工审批和失败重试。

典型例子：

- 工单分派与自动处置
- 客诉升级与协同处理
- 财务审批预分析
- 运维变更预检查与执行

### 6.3 高知识密度 Agent

Agent 需要结合长期知识、历史案例和外部文档进行决策，而不是只依赖当前对话。

典型例子：

- 客服专家辅助
- 法务/合规辅助
- 医疗流程支持系统
- 企业知识问答与决策支持

### 6.4 可学习型 Agent

Agent 需要从高频成功案例中沉淀流程模板和技能，逐步从“每次都靠推理”演化为“常规问题自动化处理”。

## 7. 需求范围定义

### 7.1 第一阶段目标

第一阶段只交付“可运行的基础认知内核”，不追求完整神经科学仿真。

第一阶段必须做到：

- 能定义 Agent
- 能执行结构化认知周期
- 能维护目标与状态
- 能调用工具与记忆
- 能做基础元认知与风险控制
- 能沉淀可复用技能
- 能回放与评估

### 7.2 不纳入第一阶段的内容

- 强依赖训练的新模型研发
- 完整多智能体社会模拟
- 复杂连续控制和机器人实时控制
- 高保真生成式世界模型训练平台
- 完整情感计算体系

## 8. 功能需求

### 8.1 Agent 定义与实例化

FR-01：系统必须支持定义 Agent Profile。

Agent Profile 至少包含：

- Agent 标识与版本
- 角色描述与领域约束
- 可用工具集
- 记忆策略
- 世界模型策略
- 安全策略
- 目标管理策略
- 默认执行预算

FR-02：系统必须支持 Agent Session 的创建、恢复、暂停和终止。

FR-03：系统必须支持同步调用与异步后台任务两种运行模式。

### 8.2 目标管理

FR-04：系统必须支持层次化 Goal Stack。

每个 Goal 需要具备：

- goal_id
- 父子关系
- 类型
- 完成条件
- 优先级
- 截止时间
- 当前状态
- 依赖关系
- 预算与权限边界

FR-05：系统必须支持目标分解、暂停、恢复、取消、重规划。

FR-06：系统必须支持“信息缺口”作为一等对象，允许 Agent 在执行前声明缺失条件并决定是继续推理、主动探索还是请求用户补充。

### 8.3 认知周期与全局工作空间

FR-07：系统必须支持标准化 Cognitive Cycle。

建议的标准周期为：

1. Perceive：接收输入和环境事件
2. Retrieve：召回相关记忆、技能和状态
3. Simulate：预测候选行动的影响
4. Deliberate：生成或选择策略
5. Gate：进行元认知、安全和预算审查
6. Act：执行工具或内部变更
7. Observe：采集执行结果
8. Learn：更新状态、记忆、技能和评估记录

FR-08：系统必须支持全局工作空间快照（Workspace Snapshot）。

Snapshot 至少包含：

- 当前目标栈
- 当前上下文摘要
- 最近观测
- 候选行动
- 记忆召回摘要
- 风险与置信度
- 执行预算
- 当前决策理由

FR-09：系统必须支持模块输出竞争与优先级仲裁，而不是简单按固定顺序拼接上下文。

### 8.4 记忆系统

FR-10：系统必须支持四层记忆：

- Working Memory：当前会话短期态
- Episodic Memory：情景记忆
- Semantic Memory：规则与稳定知识
- Procedural Memory：技能/流程模板

FR-11：系统必须支持记忆写入策略。

至少包括：

- 即时写入
- 延迟写入
- 去重
- 摘要化
- 重要性阈值
- 保留周期

FR-12：系统必须支持基于多因素的记忆检索。

检索维度至少包括：

- 语义相关度
- 时间邻近性
- 来源可信度
- 任务相似度
- 成败标签
- 风险标签

### 8.5 世界模型与预测

FR-13：系统必须支持“行动前预测”接口。

即对于候选行动，系统能够给出：

- 预期结果
- 成功概率
- 潜在副作用
- 预估成本
- 所需前置条件

FR-14：系统必须支持预测误差回写。

行动执行后，应记录：

- 预测结果
- 实际结果
- 偏差类型
- 偏差大小
- 是否影响后续策略

FR-15：第一阶段至少提供轻量世界模型能力，不要求完整概率图推断，但必须提供可插拔的模拟与评估接口。

### 8.6 推理、技能与工具执行

FR-16：系统必须将“推理模块”和“执行模块”解耦。

FR-17：系统必须支持技能模板的注册、匹配、版本化和触发。

FR-18：系统必须支持以下三类技能：

- Prompt Skill：面向推理任务的模板化能力
- Workflow Skill：面向多步执行的流程模板
- Compiled Skill：由历史经验沉淀出的结构化技能

FR-19：系统必须支持工具调用策略。

至少包括：

- 权限校验
- 幂等性标记
- 超时控制
- 重试策略
- 副作用标记
- 回滚/补偿标记

### 8.7 元认知、安全与人类介入

FR-20：系统必须支持置信度评估。

FR-21：系统必须支持冲突检测。

冲突来源至少包括：

- 推理结果与世界模型冲突
- 当前计划与权限策略冲突
- 新观测与既有记忆冲突
- 多个候选行动互斥

FR-22：系统必须支持“升级决策”。

即在低置信度、高风险、高成本或高影响场景下，系统能够：

- 请求更多信息
- 切换到深度推理
- 触发人工审批
- 执行安全降级
- 终止执行

FR-23：系统必须支持多层安全治理。

至少包含：

- 输入安全检查
- 工具权限控制
- 输出约束
- 敏感数据处理
- 审计日志

### 8.8 观测、调试与评估

FR-24：系统必须支持完整 Trace。

Trace 至少包含：

- 每个认知周期的快照
- 每个模块的输入输出摘要
- 决策依据
- 工具调用结果
- 记忆读写记录
- 策略切换原因

FR-25：系统必须支持会话回放。

FR-26：系统必须支持离线评估与在线评估。

FR-27：系统必须支持版本对比。

开发者应能够对比不同模型、不同策略、不同技能版本在相同任务集上的效果。

## 9. 非功能需求

### 9.1 可用性与稳定性

- 支持任务级重试与恢复
- 支持检查点持久化
- 支持幂等执行
- 支持模块级降级
- 避免单个工具或模型失败导致整个会话崩溃

### 9.2 性能

- 简单问答场景允许走快速路径，不强制启用全认知周期
- 复杂任务支持并行检索、并行模拟和异步执行
- 支持预算驱动的资源分配
- 支持 Token、时延、成本的细粒度度量

### 9.3 扩展性

- 模型接入可替换
- 存储后端可替换
- 模块策略可扩展
- 工具协议可扩展
- 部署模式可扩展

### 9.4 可解释性

- 每个关键行动必须可追溯到目标、输入、策略、风险和决策依据
- 系统必须支持输出“为什么执行这个动作，而不是另一个动作”

### 9.5 多租户与安全

- 租户级配置隔离
- Agent 级密钥与权限隔离
- 敏感数据脱敏与审计
- 支持企业合规接入

## 10. 版本范围建议

### 10.1 MVP 必须交付

MVP 建议聚焦以下内核：

1. Agent Profile + Session 管理
2. Goal Stack
3. Workspace Snapshot
4. LLM 推理适配层
5. 工具注册与执行网关
6. Working/Episodic/Procedural 三层记忆
7. 基础元认知网关
8. Trace/Replay/Eval

### 10.2 V1 增强能力

1. 语义记忆离线巩固
2. 技能自动提炼
3. 轻量世界模型预测
4. 多策略路由
5. 企业级多租户治理

### 10.3 V2 演进能力

1. 更强的主动推断与规划
2. 多 Agent 协作协议
3. 跨会话持续学习
4. 更高保真的状态模拟
5. 领域化认知插件市场

## 11. 验收指标

### 11.1 效果指标

- 长任务完成率
- 工具调用成功率
- 失败后恢复成功率
- 人工接管率
- 计划重排成功率
- 同类任务复用技能后的成本下降比例

### 11.2 工程指标

- 平均任务时延
- 平均 Token 消耗
- 单任务平均工具调用数
- 异常率
- 回放还原率
- 模块级可观测覆盖率

### 11.3 业务指标

- 复杂任务自动化比例
- 人工审核工作量下降比例
- 任务交付稳定性提升
- 复用率与沉淀率

## 12. 风险与关键决策

### 12.1 主要风险

1. 范式正确，但首版过度复杂，导致无法落地。
2. 世界模型能力过重，拖慢整体交付。
3. 模块太多但接口不清，反而引入新的不可控复杂度。
4. 观测和评估不足，导致系统不可调试。
5. 把“神经科学类比”误当作必须等比例实现的工程要求。

### 12.2 关键决策

1. 先做“工程可运行的认知内核”，再做高保真认知模拟。
2. 先定义统一协议与状态模型，再扩展算法能力。
3. 先把元认知、安全、回放做进内核，而不是后补。
4. 世界模型第一阶段做“可插拔预测器”，不是“重训练平台”。
5. SDK 与 Runtime 必须使用同一套协议和数据模型。

## 13. 结论

NeuroCore 的首要任务不是证明“类脑架构”在理论上成立，而是把它转化成一个企业可采纳、开发者可集成、平台可治理的 Agent 基础设施。

因此，第一步需求方案的结论是：

- 产品定位应是 Agent 认知运行时，而不是提示词框架
- 对外形态应同时覆盖 SDK 与 Runtime
- 第一阶段应聚焦目标管理、认知周期、记忆、技能、元认知、工具执行与可观测
- 世界模型与主动推断能力应采用分阶段演进策略

这是后续设计方案和架构设计的基础约束。
